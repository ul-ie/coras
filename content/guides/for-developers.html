About This System

The Córas Design System is developed using 11ty (Eleventy), a framework-agnostic and customisable static site generator.

Although this platform is node/npm-based, JavaScript versions of both Twig and YML are implemented to ensure the component code integrates with Drupal as seamlessly as possible.

Pre-Requisities

To develop components you need to be familiar with the following:

  • Single Directory Components
  • Tailwind CSS

Single Directory Components (SDC) are part of Drupal core as of version 10.3. With SDC all the files necessary to render a component are housed in a single directory named components, on the same level as a theme's templates directory.

For CSS, we are not using the Drupal BEM methodology, but are implementing a customised Tailwind configuration so as to better standardise our styling.

Using both SDC and Tailwind together, we aim to promote consistency, reusability and robustness, and improve the scalability and maintainability of our codebase.

Development Process

Components are to be built inside of Drupal; there will be no need to contribute components to the design system directly. This is the job of UL's design system developer.

The overall development process can be summed up as follows:

  • Designers hand over completed Figma components and page designs to developers.
  • These are developed as components and/or templates in the new UL Drupal theme.
  • Functionality and visual fidelity are reviewed by the design and development team in a deployed development environment.
  • Code is reviewed to ensure it follows design system standards.
  • Completed components can be integrated into design system and documented.

Note: At a later point (post re-design) we will look at making the components in the design system directly importable to the Drupal theme. If we attempt this too soon, it could become an unnecessary development bottleneck.

Figma Designs

Our visual design team manages the Figma component library and individual page wireframes (which can be forwarded on to the development team as/when needed).

Care has been taken to include variants (e.g. colour, size), states (e.g. hover, active), and mobile/desktop views to eliminate for the need for developer guesswork. However, if you do encounter any gaps in the design of our components or wireframes, please bring them to the design team rather than make design decisions yourself to address these gaps.

Local Development

Local development environments should use:

  • DDEV
  • Browsersync (for DDEV)
  • Tailwind

An example repository can be provided with the above already installed and configured, including custom DDEV commands.

Theme Development

As previously mentioned, the new UL Drupal theme is to be split into a components directory and a templates directory. Some component-related functions will rely on a custom module (details below).

Module Development

In the example repository/reference theme, a custom module called UL Twig Tweaks is provided to replicate some of the JavaScript functions used in Córas for pulling in and parsing the icon JSON data.

Certain features, such as the integration of a new search system, may require dedicated custom module development and/or integration of an external API.

It may also be necessary to update code in existing custom modules to ensure compatibility with the new theme.

Remote Environments

An always-available and performant development environment should be provided to the design and development teams to inspect and test work in progress.

Component Development

Because Single Directory Components are a new feature of Drupal, even experienced Drupal developers may need some guidance in this area. We also want to ensure consistency from the outset. To that end, we have some guidelines and example components already built.

YML Files

Each component must have both a *.twig file and a *.component.yml file. Although it is not currently a Drupal requirement to have a fully annotated yml file for a component to function, we want to ensure our components are as future-proofed and as well-documented as possible. It is, therefore, a requirement from our point of view.

For an example of a fully annotated yml file, see the below icon.component.yml file (also available via the icon component page).

The yml file essentially describes the API of the component, i.e. the publicly available properties that a template or other component can call/consume.

{{ html(get_file("inc/components/icon/icon.component.yml")) }}

Namespacing

When naming properties, use either the full component name as a prefix (e.g. icon_*), or an abbreviation of the component name (e.g. btn_*). This essentially namespaces the component properties, protecting them from unintentional modification when included in templates with their own set of variables.

Validation

Although the yml component files can be set up to validate properties, it is recommended to include additional validation in the twig template. For examples, here's how the icon_type property should be coded to ensure the enumerated values in the yml file are adhered to:

The yml property:


  icon_type:
    title: icon_type
    type: string
    description: The type (or style) of the chosen icon. This must be correct as it determines part of the file path.
    enum:
      - brands
      - duotone
      - regular
      - solid
    default: solid

The equivalent twig variable:

{{ html("
  {% set icon_type = icon_type in ['brands', 'duotone', 'regular', 'solid'] ? icon_type : 'solid' %}
  ")}}

Styling

As previously mentioned, we use Tailwind to style our components, which should eliminate the need for writing additional CSS files for both components and templates.

However if Tailwind classes are not well managed, i.e. if they are all dumped together indiscriminately without first organising them, they can in some cases become unwieldly. We therefore recommend that classes are grouped logically and then merged in the actual component. This makes styling easier to maintain and keeps the component markup as clean and as readable as possible.

{{ html('
  
  ')}}

  {% set core_classes = "manageable-list-of-classes-grouped-sensibly" %}

    core_classes,
    size_classes,
    ... etc
  ]|join(" ") %}

  
  ')}}
